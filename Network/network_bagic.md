# 🌐 네트워크 - 기초

## 📌 1. 네트워크 및 통신망 구조

### 네트워크 (Network)
* **정의**: 컴퓨터, 스마트폰 등의 통신 기기들이 서로 연결되어 데이터를 원활하게 주고받을 수 있도록 구성된 통신망

### LAN과 WAN
* **LAN (Local Area Network)**: 건물 안이나 특정 지역 등 비교적 좁은 범위를 커버하는 근거리 네트워크
* **WAN (Wide Area Network)**: 지리적으로 멀리 떨어진 2개 이상의 LAN을 서로 연결한 광역 네트워크

### LAN의 구성 방법 (토폴로지)
* **스타형**: 하나의 중앙 허브(또는 스위치)에 여러 대의 컴퓨터를 방사형으로 연결하는 방식
* **링형**: 개별 컴퓨터가 서로 원형(Ring)으로 연결되어 한쪽 방향으로 데이터를 전송하는 구성
* **버스형**: 하나의 긴 중심 케이블(공통 회선)에 컴퓨터를 포함한 모든 주변 장치를 연결하는 방식

---

## 🛠️ 2. 네트워크 핵심 장비

### 리피터와 허브 (1계층 장비)
* **리피터 (Repeater)**: 거리가 멀어지면서 감쇠된 전기적 신호를 증폭·복원하여 통신 거리를 연장하는 장비
* **허브 (Hub)**: 리피터 기능을 포함하며, 여러 대의 컴퓨터를 하나의 네트워크로 연결할 수 있도록 포트를 여러 개 제공하는 다중 포트 리피터 (모든 포트에 데이터를 뿌리는 플러딩 방식 사용)

### 스위치 (2계층 장비)
* 허브와 외형은 비슷하지만, 목적지 **MAC 주소**를 학습(`MAC Address Table`)하여 필요한 포트로만 프레임을 정확히 전달하는 장비
* **장점**: 대역폭을 효율적으로 사용하고, 데이터 충돌(Collision)을 획기적으로 줄임

### 라우터 (3계층 장비)
* 컴퓨터 네트워크에서 데이터를 목적지까지 가장 안전하고 빠르게 전달하는 네트워크 중계 장비
* **IP 주소**를 기반으로 최적의 경로를 선택하는 라우팅 기능을 수행함

---

## 🔢 3. 데이터 단위 및 통신 규칙

### 비트와 바이트
* 컴퓨터가 데이터를 처리하고 저장하는 정보량의 단위
* **비트 (Bit)**: $0$과 $1$을 나타내는 데이터의 최소 단위
* **바이트 (Byte)**: 8개의 비트를 하나로 묶은 단위($1 \text{ Byte} = 8 \text{ bits}$)로, 문자 하나를 표현하는 기본 단위

### 프로토콜 (Protocol)
* 컴퓨터나 네트워크 기기들이 서로 데이터를 오차 없이 원활하게 교환하기 위해 사전에 약속한 **통신 규약**

---

## 📡 4. 매체 접근 제어: CSMA/CD

### CSMA/CD 개요
* 유선 LAN(주로 구형 이더넷) 환경에서 여러 기기가 하나의 통신 회선을 공유할 때 발생하는 **데이터 충돌을 방지하고 관리**하는 매체 접근 제어(MAC) 프로토콜

### CSMA/CD 기본 과정
1. **Carrier Sense**: 데이터를 보내기 전, 케이블이 현재 사용 중인지(신호가 흐르는지) 확인
2. **Multiple Access**: 회선이 비어 있으면 데이터를 전송 (여러 기기가 동시에 접근 가능)
3. **Collision Detection**: 데이터 전송 중 또는 전송 후에 충돌이 발생했는지 감지 $\rightarrow$ 충돌 발생 시 일정 시간 대기 후 재전송

### Carrier Sense의 3가지 방식 (Persistence 종류)
* **1-persistent**: 채널이 비어있는지 계속 감지하다가, 비는 순간 **즉시(100% 확률로) 전송**. 만약 채널이 바쁘면 계속 대기함 $\rightarrow$ 회선이 비는 순간 대기하던 노드들이 동시에 전송하여 **충돌 확률이 매우 높음**
* **non-persistent**: 채널이 비어있는지 확인 후, 바쁘면 **임의의 시간(random backoff) 동안 기다렸다가** 다시 감지함 $\rightarrow$ 계속 감지하지 않으므로 충돌 확률은 낮아지지만, 대기 시간이 늘어나 지연이 증가할 수 있음
* **p-persistent**: 채널이 비면 **확률 $p$로 데이터를 전송**하고, $(1-p)$의 확률로 다음 슬롯까지 기다렸다가 다시 $p$ 확률로 시도함 $\rightarrow$ 충돌과 지연 사이를 타협하는 절충 방식

## 🔍 5. 주소 변환 기술 (ARP, NAT)

### ARP (Address Resolution Protocol)
* 네트워크상에서 논리적 주소인 **IP 주소를 물리적 네트워크 주소인 MAC 주소로 변환(매핑)**해 주는 프로토콜

#### 🔄 ARP 작동 방식 (2단계)
1. **ARP Request (요청)**: 목적지 IP는 알지만 MAC 주소를 모를 때, 동일 네트워크의 모든 장치에게 **브로드캐스트**로 MAC 주소를 요청함
2. **ARP Reply (응답)**: 해당 IP를 가진 기기가 요청한 장치에게 자신의 MAC 주소를 **유니캐스트(1:1)**로 직접 응답함

### NAT (Network Address Translation)
* 네트워크 트래픽이 라우터나 방화벽을 통과할 때, IP 패킷의 출발지/목적지 IP 주소 및 포트 번호를 변환하는 기술 (주로 사설 IP $\leftrightarrow$ 공인 IP 변환에 사용)
* **장점**: 
  * **IP 주소 절약**: 제한된 공인 IPv4 주소를 효율적으로 사용 가능
  * **보안 강화**: 내부 사설 IP 주소를 외부에 숨길 수 있음
* **단점**: 외부 네트워크에서 내부로 먼저 통신을 시도할 때 내부 IP를 알 수 없어 **직접 연결이 불가능**함 $\rightarrow$ 이를 해결하기 위해 포트 포워딩이 필요함

### 포트 포워딩 (Port Forwarding)
* 공유기(라우터)가 외부 인터넷에서 들어오는 특정 포트의 요청을 내부 네트워크의 특정 기기(IP)와 포트로 직접 전달해 주는 설정 기능
* *예시: 외부에서 `공인IP:8080`으로 접속 $\rightarrow$ 내부 사설망의 `192.168.0.10:80`(웹 서버)으로 자동 매핑 및 전달*

## 🔀 6. 데이터 전송 방식 및 서브넷팅

### 데이터 전송 방식 비교
* **유니캐스트 (Unicast)**: 특정 **1대 (1:1)**에게만 데이터를 전송 (목적지 = 수신 호스트의 MAC 주소)
* **브로드캐스트 (Broadcast)**: 동일 네트워크 도메인 내의 **모든 호스트**에게 데이터를 전송 (목적지 MAC = `FF:FF:FF:FF:FF:FF`)
* **멀티캐스트 (Multicast)**: 특정 **그룹 (1:N)**에게만 데이터를 전송 (그룹에 가입한 호스트만 수신, 목적지 IP = `224.0.0.0` ~ `239.255.255.255`)

### 서브넷과 서브넷팅
* **서브넷 (Subnet)**: 하나의 거대한 네트워크를 효율적으로 관리하기 위해 분할한 작은 네트워크 영역
* **서브넷팅의 원리**: IP 주소 구조(`네트워크 ID` + `호스트 ID`)에서 네트워크 ID는 그대로 두고, **호스트 ID 영역을 쪼개어** 서브넷 ID로 활용하는 방식

### 서브넷 마스크 (Subnet Mask)
* IP 주소에서 어디까지가 네트워크 영역이고 어디서부터가 호스트(기기) 영역인지 구분해 주는 32비트 숫자
* *예시: `192.168.10.0/24` $\rightarrow$ 앞의 24비트인 `192.168.10`까지가 네트워크 영역이고, 나머지 호스트 영역을 구분함*

---

## 🗺️ 7. 라우팅 (Routing)

### 라우터의 역할
* 데이터 전송을 위해 목적지까지 가장 빠르고 효율적인 길을 찾아 네트워크 간의 경로를 설정하고 패킷을 중계함

### 라우팅 테이블 (Routing Table)
* 라우터가 목적지 네트워크로 가기 위한 **최적의 경로 정보, 거리, 인터페이스, 비용(Metric)** 등을 기록해 둔 지도와 같은 표

---

## ⚡ 8. 전송 계층 프로토콜: TCP / UDP

### TCP (Transmission Control Protocol)
* **신뢰성 중심**: 데이터가 누락 없이 정확하게, 보낸 순서대로 손실 없이 도착하는 것을 보장
* **연결 설정**: 데이터를 본격적으로 보내기 전, 송수신자 간에 논리적인 연결을 먼저 수립 (**3-Way Handshake**)
* **흐름 및 혼잡 제어**: 수신자의 처리 속도보다 데이터가 너무 많이 오거나 네트워크가 혼잡해지는 것을 감지하여 전송 속도를 스스로 조절

### UDP (User Datagram Protocol)
* **속도 중심**: 연결 수립 과정이나 패킷 검증/재전송 같은 신뢰성 보장 메커니즘을 포기하는 대신, 오버헤드가 적어 전송 속도가 매우 빠름
* **데이터그램 방식**: 수신자의 상태를 전혀 확인하지 않고 일방적으로 데이터를 밀어 넣는 방식
* **실시간 통신에 유리**: 실시간 스트리밍, 온라인 게임, 화상 통화 등 일부 데이터가 유실(화면 깨짐, 소리 끊김)되더라도 지연 없이 다음 데이터를 계속 보내는 것이 더 중요한 곳에 적합

---

## 🤝 9. TCP 3-Way & 4-Way Handshake

### TCP 3-Way Handshake (연결 수립 과정)
* **정의**: 통신을 시작하기 전, 두 기기 간에 안전하고 논리적인 연결을 맺는 과정

1. **1단계 [SYN]**: 클라이언트가 서버에 연결 요청을 위해 `SYN`(접속 요청 플래그)을 전송 *(클라이언트 상태: `SYN_SENT`)*
2. **2단계 [SYN + ACK]**: 서버가 클라이언트의 요청을 허가한다는 의미의 `ACK`(응답)와, 서버 역시 클라이언트와 연결을 맺고 싶다는 `SYN`을 함께 묶어서 전송 *(서버 상태: `SYN_RCVD`)*
3. **3단계 [ACK]**: 클라이언트가 서버의 수락을 확인하고 최종적으로 확인 응답인 `ACK`를 보냄. 서버가 이를 받으면 최종적으로 연결이 성립 *(상태: `ESTABLISHED`)*

### TCP 4-Way Handshake (연결 해제 과정)
* **정의**: 데이터 전송을 모두 마친 후, 양쪽의 연결을 안전하고 깔끔하게 종료하는 과정

1. **1단계 [FIN]**: 클라이언트가 대화를 끝내고 연결을 종료하겠다는 `FIN` 플래그를 서버에 전송 *(클라이언트 상태: `FIN_WAIT_1`)*
2. **2단계 [ACK]**: 서버가 클라이언트의 `FIN`을 받았다는 확인 응답 `ACK`를 전송. 이때 서버는 아직 보내지 못한 남은 데이터를 모두 처리할 때까지 대기하며, 클라이언트도 전송이 완전히 끝날 때까지 대기 *(서버 상태: `CLOSE_WAIT`, 클라이언트 상태: `FIN_WAIT_2`)*
3. **3단계 [FIN]**: 서버 역시 데이터 전송을 마치고 종료 준비가 완료되면 클라이언트에게 `FIN` 플래그를 보냄 *(서버 상태: `LAST_ACK`)*
4. **4단계 [ACK]**: 클라이언트가 서버의 `FIN`을 받고 최종 확인 응답인 `ACK`를 전송. 서버는 연결을 즉시 해제하며, 클라이언트는 혹시 나중에 도착할지 모르는 패킷 유실을 대비해 일정 시간 대기(`TIME_WAIT`)한 후 완전히 연결을 종료

## 🖥️ 10. 주요 네트워크 서버 및 프로토콜

### DNS (Domain Name System) 서버
* **정의**: 사람이 기억하기 쉬운 도메인 이름(예: `naver.com`)을 컴퓨터가 이해할 수 있는 IP 주소(예: `223.130.195.200`)로 변환해 주는 시스템
* **작동 원리**: 브라우저에 URL을 입력하면 DNS 서버에 해당 도메인의 IP를 조회한 뒤, 반환받은 IP 주소를 통해 실제 웹 서버에 접속함

### DHCP (Dynamic Host Configuration Protocol) 서버
* **정의**: 네트워크 안에 있는 기기들에게 IP 주소, 서브넷 마스크, 기본 게이트웨이 등의 네트워크 설정 정보를 자동으로 할당해 주는 프로토콜
* **장점**: 관리가 편해지고 고정 IP 충돌 문제를 방지할 수 있으며, 기기가 네트워크를 이탈하면 해당 IP를 회수해 다른 기기에 재할당하므로 IP 자원을 효율적으로 사용 가능

### SMTP & POP3 프로토콜 (이메일 프로토콜)
* **SMTP (Simple Mail Transfer Protocol)**: 클라이언트가 작성한 메일을 메일 서버로 보내거나, 메일 서버끼리 메일을 보낼 때 사용하는 **송신 프로토콜**
* **POP3 (Post Office Protocol 3)**: 메일 서버에 수신되어 저장된 이메일을 클라이언트 기기로 다운로드하여 읽을 수 있도록 해주는 **수신 프로토콜** (메일을 가져오면 기본적으로 서버에서 삭제됨)

## 📦 11. OSI 7계층 데이터 흐름: 캡슐화 및 역캡슐화



### 캡슐화 (Encapsulation) - 송신 프로세스
* 데이터가 상위 계층에서 하위 계층으로 내려가면서 각 계층의 특성 정보(제어 정보, 주소 등)가 담긴 **헤더(Header)**를 붙여나가는 과정
1. **Application / Presentation / Session**: 사용자가 작성한 순수 데이터 생성 (`Data`)
2. **Transport**: 데이터에 포트 번호와 시퀀스 번호 정보가 담긴 TCP 헤더를 추가하여 세그먼트 생성 (`Segment`)
3. **Network**: 세그먼트에 출발지/목적지 IP 주소가 담긴 IP 헤더를 추가하여 패킷 생성 (`Packet`)
4. **Data Link**: 패킷에 물리적 주소인 MAC 주소 헤더와 에러 검출용 트레일러(FCS)를 추가하여 프레임 생성 (`Frame`)
5. **Physical**: 프레임을 전기적/광학적 신호인 비트로 변환하여 물리 매체로 전송 (`Bit`)

### 역캡슐화 (Decapsulation) - 수신 프로세스
* 데이터가 하위 계층에서 상위 계층으로 올라가면서 수신한 데이터의 **헤더를 하나씩 벗겨내며** 상위 계층으로 전달하는 과정
1. **Physical**: 물리 매체로 들어온 비트 신호를 전기 신호로 받아 데이터 링크 계층으로 전달
2. **Data Link**: MAC 주소를 확인해 본인에게 온 것이 맞으면 데이터 링크 헤더와 트레일러를 제거하고 네트워크 계층으로 전달
3. **Network**: IP 주소를 확인해 목적지가 맞으면 IP 헤더를 제거하고 전송 계층으로 전달
4. **Transport**: 포트 번호를 확인해 해당 목적지 애플리케이션을 찾아 TCP 헤더를 제거하고 데이터를 전달
5. **Session / Presentation / Application**: 최종적으로 헤더가 모두 제거된 순수 데이터를 사용자가 프로그램을 통해 확인

## ⚖️ 12. 로드밸런서 (Load Balancer)

### 로드밸런서 개요
* **정의**: 여러 대의 서버를 두고, 사용자의 요청(트래픽)이 한쪽 서버로 몰리지 않도록 고르게 분산시켜 주는 네트워크 장치
* **인프라 확장 방식 (Scale Out)**: 서버의 성능 자체를 높이는 것(Scale Up)이 아니라, 비슷한 사양의 서버를 여러 대로 늘려 부하를 분산시키는 구조에 필수적

### L4 로드밸런서 (전송 계층)
* **특징**: **IP 주소와 포트 번호**를 기준으로 트래픽을 분산시킴. 데이터의 내용을 열어보지 않기 때문에 속도가 빠르고 효율적임

#### 🔄 로드밸런싱 알고리즘 방식
* **라운드로빈 (Round Robin)**: 가장 기본적인 방식으로, 들어오는 요청을 순서대로 서버별로 돌아가면서 균등하게 배분
* **가중 라운드로빈 (Weighted Round Robin)**: 라운드로빈과 같지만 서버의 처리 능력에 따라 가중치를 부여하고, 성능이 더 좋은 일부 서버에 큰 트래픽을 몰아 받는 방식
* **랜덤 (Random)**: 특별한 규칙 없이 무작위로 서버를 선택해 분배하는 방식
* **해시 (Hash)**: 특정 클라이언트는 특정 서버에만 처리하도록 고정하는 방식 (세션 유지에 유용)

### L7 로드밸런서 (응용 계층)
* **특징**: 전송 계층에서의 로드밸런서가 IP나 포트 번호를 기준으로 부하를 분산시켰다면, 응용 계층의 로드밸런서는 HTTP 헤더, 쿠키, URL 데이터 내용 등 애플리케이션 영역의 메시지를 직접 읽어 **서비스별(URL 경로별)로 부하 분산**을 진행함

## 🔒 13. VPN (Virtual Private Network)

### VPN (가상 사설망)
* **정의**: 인터넷을 통해 데이터를 안전하게 전송하기 위한 기술
* **특징**: 공용 인터넷 망을 마치 전용 사설망처럼 사용하여 데이터를 암호화하고 외부의 위변조나 도청으로부터 안전하게 보호함
